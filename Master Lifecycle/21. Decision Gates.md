# Decision Gates

This document is the **canonical registry** of decision gates for the Master Lifecycle. Every numbered phase document (`07.` through `20.`) references gates by the IDs defined here. Where the legacy **`Universal Software Project Development Procedure.md`** uses a different gate numbering, the cross-reference table in §3 reconciles the two.

For lifecycle context see `06. Lifecycle Overview.md`; for the artifact register that supplies gate evidence see `22. Required Documents.md`.

---

## 1. Purpose

Decision gates are the points at which a project is allowed to **continue, change direction, pause, or stop**. Without gates the lifecycle is descriptive; with gates it is governed. A project shall not advance from one phase to the next unless the relevant gate has produced:

- a recorded **decision**,
- supporting **evidence**,
- an identified **decision authority**,
- a clear **next action** if the gate fails.

---

## 2. Master Lifecycle Gates (canonical — Gates G1–G10)

The Master Lifecycle defines ten gates aligned 1:1 with phase exits. Phases 12–14 (Deployment, Maintenance, Post-Release Review) share Gate G9 / G10 because their checkpoints are operational milestones rather than entry/exit reviews.

| Gate | Name | Asked at end of | Anchors entry to |
| --- | --- | --- | --- |
| **G1** | Idea Accepted for Analysis | Phase 1 — Idea Capture | Phase 2 |
| **G2** | Problem Validated | Phase 2 — Problem Definition | Phase 3 |
| **G3** | Project Selected | Phase 3 — Project Evaluation and Selection | Phase 4 |
| **G4** | Requirements Approved | Phase 5 — Requirements Definition | Phase 6 |
| **G5** | Architecture Approved | Phase 7 — Architecture and Design | Phase 8 |
| **G6** | Development Ready | Phase 8 — Development Preparation | Phase 9 |
| **G7** | Testing Passed | Phase 10 — Testing and Validation | Phase 11 |
| **G8** | Release Approved | Phase 11 — Release Preparation | Phase 12 |
| **G9** | Deployment Completed | Phase 12 — Deployment | Phase 13 |
| **G10** | Maintenance Review Completed | Phase 14 — Post-Release Review | next planning cycle |

**Note on Phase 4 (Feasibility and Business Case):** the feasibility decision is treated as part of **G3** rather than a separate gate, because selection (Phase 3) and feasibility (Phase 4) jointly determine whether the project is funded. Some programs split this into G3a (Selection) and G3b (Funded Business Case); record the variant in the project's governance plan if used.

## 3. Cross-reference: Universal Procedure gates → Master Lifecycle gates

The legacy `Universal Software Project Development Procedure.md` uses a compressed 7-gate scheme (UG1–UG7). When reading a project that cites those numbers, use this map.

| Universal Procedure | Name in Universal Procedure | Maps to Master Lifecycle Gate(s) |
| --- | --- | --- |
| **UG1** | Selection | **G3** (and incorporates G1, G2 evidence) |
| **UG2** | Scope and requirements | **G4** |
| **UG3** | Design | **G5** |
| **UG4** | Planning | **G6** (and aspects of G4 when planning is split out) |
| **UG5** | Implementation readiness | **G6** (Development Ready) |
| **UG6** | Release | **G7** + **G8** (Testing Passed and Release Approved) |
| **UG7** | Maintenance | **G9** + **G10** |

**Authority:** when the Master Lifecycle and the Universal Procedure conflict, the **Master Lifecycle Gates (G1–G10)** are canonical for new and active projects. The Universal Procedure remains valid as a reusable backbone but its gate numbers should be cited as **UG1–UG7** to avoid collision.

## 4. Gate Specification (every gate must answer four questions)

Each gate, wherever it appears in a phase document, shall answer:

1. **Can the project move forward?** (Yes / Conditionally / No / Defer)
2. **What evidence proves it?** (named artifacts, with versions or dates)
3. **Who approves it?** (named authority and role)
4. **What happens if it fails?** (return to which phase, what to remediate, when to re-review)

This is the minimum evidence shape; richer programs may add risk register updates, compliance attestations, or audit log entries (`cybercube-audit-directives.mdc`).

For **G3 / G4 / G5 / G6 / G8 / G10**, required evidence includes applicable CYBERCUBE standard evidence from `25. Quality and Compliance Checks.md` §5 and `31. Appendix D — CYBERCUBE Standards Applicability Matrix.md`, or a documented non-applicability rationale.

## 5. Gate Catalog

### G1 — Idea Accepted for Analysis

- **Phase exit:** Phase 1 — Idea Capture
- **Required evidence:** Idea Capture Form (Template A-0)
- **Authority:** Standards / Governance Reviewer (or Project Owner for solo projects)
- **Possible decisions:** Accepted for Problem Definition · Deferred · Rejected · Research More
- **On failure:** retain the form for traceability; if Research More, return to sponsor with specific gaps listed

### G2 — Problem Validated

- **Phase exit:** Phase 2 — Problem Definition
- **Required evidence:** Problem Definition Document (Template A-0.1) with at least one independent validation source
- **Authority:** Product / Business Reviewer + Standards / Governance Reviewer
- **Possible decisions:** Validated — Proceed to Selection · Return to Phase 1 · Deferred · Rejected
- **On failure:** return to Phase 1 if scope was wrong, defer if timing was wrong, reject if no real problem exists

### G3 — Project Selected

- **Phase exits:** Phase 3 — Project Evaluation and Selection (selection signal) and Phase 4 — Feasibility and Business Case (funding signal)
- **Required evidence:** Project Selection Scorecard / project evaluation record (Template A-3.1), Business Field Report (Template A-4) or approved waiver, Feasibility Assessment (Template A-3.2), Business Case (Template A-3.3), confirmed complexity level or explicit Unknown resolution plan per `23. Project Complexity Levels.md`, and PRCS candidate metadata where the initiative may become a product (`PRD-XXX` if assigned, candidate `PCL-L.D.E.C`, domain tags, function descriptors, or Work Type Tag rationale if pre-product/cross-cutting)
- **Authority:** Product / Business Owner + sponsor with funding authority
- **Possible decisions:** Proceed to Requirements · Pivot · Defer · Stop
- **On failure:** record rationale and revisit triggers; deferred candidates remain in the portfolio backlog

### G4 — Requirements Approved

- **Phase exit:** Phase 5 — Requirements Definition
- **Required evidence:** Requirements Specification Package (Template A-8), including baselined CRS (Template A-1), SRS (Template A-2), NFR inputs (Template A-10 / NFR-001 where applicable), traceability seeds, and evidence that requirements depth matches the confirmed complexity level or Unknown resolution plan
- **Authority:** Product Owner + Engineering Lead
- **Possible decisions:** Approved · Approved with conditions · Returned for revision
- **On failure:** scope back into Phase 5 with the unresolved items listed; re-baseline before re-approval

### G5 — Architecture Approved

- **Phase exit:** Phase 7 — Architecture and Design
- **Required evidence:** Architecture Design Document (**ARD-001**), ADRs, Data Model / ERD Document (Template A-11, when data modeling is in scope), API and Integration Contract (Template A-12, when APIs/integrations are in scope), UI/UX Design Document (**UXD-001**, when UI/UX is in scope), and Module/Component Design Documents where required
- **Authority:** Architecture Review Board (ARB) per `cybercube-architecture-directives.mdc`
- **Possible decisions:** Approved · Approved with deviations (logged) · Returned for revision
- **On failure:** open ADR(s) for unresolved decisions; loop back to Phase 7 sections affected

### G6 — Development Ready

- **Phase exit:** Phase 8 — Development Preparation
- **Required evidence:** Environment and Delivery Strategy (Template A-14), Module and File Planning Document (Template A-13), Development Plan (Template A-15) with selected execution decomposition model recorded (Development Plan only, TD-001, HG-001, or approved alternative), Development Setup Guide, Coding Standards Reference, Naming/Identifier Conformance Notes for `STD-ENG-001` (Namespace M naming, lifecycle-local ID policy, Namespace G mappings where applicable, and CC-PID/raw database ID rules where public IDs exist), and Initial Test Setup (per `14. Phase 8 — Development Preparation.md`; setup/coding/test evidence may be standalone or controlled sections inside the Phase 8 templates)
- **Authority:** Engineering Lead + Tech Lead
- **Possible decisions:** Ready · Ready with caveats · Not ready
- **On failure:** identify which preparation artifact is missing; do not start Phase 9 until remediated

### G7 — Testing Passed

- **Phase exit:** Phase 10 — Testing and Validation
- **Required evidence:** Test Strategy execution results (Template A-16), Acceptance Test Scenarios outcomes (Template A-17), QA Results (Template A-18), Bug Reports with severity and disposition (Template A-19, when defects exist), and Validation Sign-Off (Template A-20)
- **Authority:** QA Lead + Product Owner
- **Possible decisions:** Passed · Passed with known issues (logged) · Failed
- **On failure:** loop back to Phase 9 for remediation, then re-test; severity gating per program policy

### G8 — Release Approved

- **Phase exit:** Phase 11 — Release Preparation
- **Required evidence:** Release Readiness Checklist (Template A-21), Known Issues List (Template A-22, when issues remain open or deferred), Release Notes (Template A-23), Rollback Plan (Template A-24), Security Review Results (Template A-25, when required; supported by Security Review Checklist Template A-25.1), Final Approval Record (Template A-26), PRCS production confirmation for product releases (`PRD-XXX`, approved `PCL-L.D.E.C`, domain tags, owner, and criticality-driven downstream standard tiering), and Naming/Identifier release confirmation where public IDs, governance IDs, modules/components/files, APIs, logs, or release artifacts are in scope
- **Authority:** Release Manager + Sponsor
- **Possible decisions:** Approved · Approved with conditions · Held
- **On failure:** record blockers; do not deploy until blockers are cleared

### G9 — Deployment Completed

- **Phase exit:** Phase 12 — Deployment
- **Required evidence:** Deployment Checklist signed (Template A-27), Deployment Record (Template A-28), Migration Record (Template A-29, when migrations are in scope), Environment Configuration Record (Template A-30), and Rollback Confirmation or non-invocation note (Template A-31)
- **Authority:** Release Manager + Operations Lead
- **Possible decisions:** Completed · Completed with rollback · Failed
- **On failure:** invoke rollback per Rollback Plan; trigger Incident Response per `cybercube-ir-directives.mdc` if appropriate

### G10 — Maintenance Review Completed

- **Phase exit:** Phase 14 — Post-Release Review (informed by Phase 13 operational data)
- **Required evidence:** Post-Release Review (Template A-38), Lessons Learned Record (Template A-39), Improvement Recommendations (Template A-40), Maintenance Backlog (Template A-35), and Process Improvement Notes (Template A-41, when process changes are identified)
- **Authority:** Product Owner + Engineering Lead + Operations Lead
- **Possible decisions:** Closed · Closed with follow-up actions · Re-open prior phase
- **On failure / follow-up:** route improvements through `27. Process Improvement.md` and `26. Change Control.md` as appropriate

## 6. Failure Routing Quick Reference

| If gate fails at | Default return phase | Notes |
| --- | --- | --- |
| G1 | sponsor (no phase) | Idea is rejected or returned with notes |
| G2 | Phase 1 | Often the idea was solution-shaped, not problem-shaped |
| G3 | Phase 3 or Phase 4 | Distinguish selection failure from funding failure |
| G4 | Phase 5 | Track unresolved requirements as blockers |
| G5 | Phase 7 sub-section | Use ADRs to scope the loopback |
| G6 | Phase 8 sub-section | Identify missing preparation artifact |
| G7 | Phase 9 | Use Bug Reports to scope the loopback |
| G8 | Phase 10 or Phase 11 | Distinguish quality blockers from release blockers |
| G9 | Rollback per Phase 12 plan | Treat as incident if customer-affecting |
| G10 | Phase 13 / 14 | Route improvements via Process Improvement |

## 7. Relationship to Other Standards

- **Audit:** every gate decision is an auditable event per `cybercube-audit-directives.mdc` §4 (findings & classification) — record decision, evidence, authority, date, rationale.
- **Change control:** scope-affecting decisions at G3, G4, G5, G8, and deployment/maintenance follow-up from G9/G10 must flow through `26. Change Control.md` where they change approved scope, release posture, or operational commitments.
- **Risk:** G3, G5, G8, and material G9/G10 follow-up require risk register updates per `cybercube-erm-directives.mdc`.
- **Architecture:** G5 follows the ARB process per `cybercube-architecture-directives.mdc`.
- **Release readiness:** G8 may incorporate the **CYBERCUBE compliance matrix** (≥40 release-grade per `cybercube-compliance-matrix.mdc`) when product classification (PRCS) requires it.
