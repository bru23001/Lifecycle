# Appendix D — CYBERCUBE Standards Applicability Matrix

## 1. Purpose

This appendix is the reusable evidence form for determining which CYBERCUBE standards apply to a project, why they apply, what evidence is required, and which lifecycle gates must confirm that evidence.

Use this appendix with `25. Quality and Compliance Checks.md` §5. That section defines the core applicability rules; this appendix provides the project-level record.

---

## 2. Lifecycle Role

| Lifecycle point | Use |
| --- | --- |
| Phase 4 / G3 | Identify high-risk applicable standards before funding or selection proceeds. |
| Phase 5 / G4 | Convert applicable standards into requirements, NFRs, constraints, and acceptance criteria. |
| Phase 7 / G5 | Reflect standards in architecture, data, API, security, privacy, observability, UX, and integration designs. |
| Phase 8 / G6 | Prepare tools, CI, coding standards, environments, test setup, and evidence controls. |
| Phase 11 / G8 | Confirm standards evidence is complete or non-applicability rationale is approved before release. |
| Phase 14 / G10 | Review gaps, exceptions, evidence quality, and next-cycle improvements. |

## 5. Machine-Readable Evidence

When the project uses automation, dashboards, or audit tooling, maintain a machine-readable copy of the matrix in JSON or YAML.

Suggested JSON shape:

```json
{
  "project": "",
  "product_id": "",
  "pcl_code": "",
  "pcl_status": "candidate",
  "domain_tags": [],
  "function_descriptors": [],
  "work_type_tags": [],
  "last_reviewed": "",
  "standards": [
    {
      "standard": "Security Policy",
      "applies": "yes",
      "rationale": "",
      "phases": ["5", "7", "9", "11"],
      "gates": ["G4", "G5", "G8"],
      "evidence": [],
      "owner": "",
      "status": "planned"
    }
  ]
}
```

Allowed `applies` values: `yes`, `no`, `conditional`.

Allowed `status` values: `planned`, `in_progress`, `complete`, `waived`, `not_applicable`, `blocked`.

---

## 6. Completion Criteria

This appendix is complete when:

1. Every listed standard has an applicability value.
2. `No` and `Conditional` entries have rationale.
3. Applicable standards have named owners.
4. Evidence artifacts are named or explicitly scheduled.
5. Gate checks are acknowledged for each applicable standard.
6. Waivers, exceptions, or deviations are routed through the correct governance path.
7. The matrix is reviewed before G8 and again during G10.
