# Traceability Rules

## Canonical source

End-to-end traceability structures, matrix templates, and change-propagation expectations are defined in **USSM — Unified Software Standards Manual v1.0.md**, especially:

- **§4.5** — CRS traceability references  
- **§5.5** — SRS traceability matrix  
- **§6.6** — Design traceability  
- **§7.4.2** — Test case traceability  
- **§10.1 Annex A** — Consolidated traceability matrix templates (CRS ↔ SRS ↔ design ↔ code ↔ tests ↔ changes)

## Minimum expectations

1. **Bidirectional links** between discovery evidence, requirements, scope, design, implementation, tests, release, deployment, maintenance, and post-release improvement records.
2. **Stable identifiers** used consistently across documents, task systems, code references, tests, release records, incidents, and improvement backlogs.
3. **Gate evidence continuity:** every gate should be able to trace its decision to the artifacts listed in `21. Decision Gates.md` and `22. Required Documents.md`.
4. **Change impact:** updating a baselined artifact triggers assessment across linked rows (per USSM governance §3 and Annex A §10.1.5).

Tailor column sets per project while preserving upstream ↔ downstream linkage described in USSM.

## Lifecycle Traceability Model

Traceability should show how an idea becomes released and then improved:

```text
Idea / Problem
  -> Selection / Business Case
  -> Requirements / Scope / Features
  -> Architecture / Data / API / UI / Module Design
  -> Development Plan / Tasks / Implementation
  -> Tests / Defects / Validation
  -> Release Approval
  -> Deployment / Configuration / Migration / Rollback
  -> Maintenance / Incidents / Metrics / Backlog
  -> Post-Release Review / Improvements / Next Cycle
```

The minimum link target is the nearest meaningful upstream and downstream artifact. For example, an implementation record may link to a task, feature, requirement, design artifact, test, and defect, without repeating the entire lifecycle chain in the same row.

## Suggested Stable ID Vocabulary

Use existing project conventions where they already exist. Otherwise, these prefixes are recommended as **lifecycle-local artifact IDs**. They are not automatically CYBERCUBE Namespace G registry IDs unless a governance registry assigns them.

| Area | Suggested IDs |
| --- | --- |
| Discovery / validation | `IDEA-`, `PROB-`, `VAL-` |
| Selection / business case | `SEL-`, `FEAS-`, `BC-`, `FORECAST-` |
| Requirements / scope | `CRS-`, `SRS-FR-`, `SRS-NFR-`, `REQ-`, `SCOPE-`, `FEAT-` |
| Design / architecture | `ADR-`, `DES-`, `DATA-`, `API-`, `UX-`, `MOD-` |
| Planning / implementation | `TASK-`, `PT-`, `MILE-`, `IMPL-`, `PR-` |
| Testing / validation | `TC-`, `ATS-`, `QA-`, `BUG-`, `VALSIGN-` |
| Release / deployment | `REL-`, `RR-`, `ROLLBACK-`, `DEP-`, `MIG-`, `ENV-` |
| Maintenance / operations | `MNT-`, `INC-`, `MCR-`, `METRIC-`, `PATCH-` |
| Post-release / improvement | `PRR-`, `LESSON-`, `REC-`, `PI-` |
| Governance / decisions | `G1`–`G10`, `WAIVER-`, `RISK-`, `CR-` |

IDs should be stable after approval. If an ID changes, record the migration from old ID to new ID.

### CYBERCUBE Naming Namespace Boundaries

| Namespace | Use in lifecycle work | Rule |
| --- | --- | --- |
| Namespace A | File, branch, package, release artifact, and deliverable names | Follow the artifact-naming convention when producing controlled files or packages. |
| Namespace B | Public entity IDs in APIs, URLs, logs, support workflows, or UI | Use CC-PID; do not expose raw database primary keys externally. CC-PID is not authentication or authorization evidence. |
| Namespace G | Registered governance IDs such as `PRD-XXX`, `MOD-XXX`, and future registry-controlled IDs | Use only when assigned by the authoritative registry; do not invent sequence numbers in local lifecycle artifacts. |
| Namespace M | Source modules, components, directories, exports, and implementation files | Confirm naming in Phase 8 before Phase 9 coding and preserve it through release evidence. |

Use `../CYBERCUBE standards/[1]-CYBERCUBE-Name-Registry.md` as the authoritative lookup for registered Namespace A/B/G/M values, UUID-to-CC-PID mappings, and project registry entries.

When a lifecycle-local ID needs to be promoted into a CYBERCUBE governance registry, record the mapping from local ID to Namespace G ID and retain both references in traceability records.

## Trace Links by Phase and Gate

| Phase / Gate | Traceability expectation |
| --- | --- |
| Phase 1 / G1 | Idea Capture Form (Template A-0) links idea, sponsor, problem hypothesis, complexity estimate, and decision. |
| Phase 2 / G2 | Problem Definition Document (Template A-0.1) links problem, impacted users, validation sources, and G2 decision. |
| Phases 3–4 / G3 | Selection scorecard, feasibility, business case, and forecasts link candidate idea/problem to funding decision and assumptions. |
| Phase 5 / G4 | CRS/SRS/NFRs, acceptance criteria, and traceability seed link stakeholder needs to testable requirements. |
| Phase 6 | Scope, feature inventory, tasks, and delivery plans link approved requirements to planned work. |
| Phase 7 / G5 | Architecture, ADRs, data model, API contract, UI/UX, and module design link requirements/features to design decisions. |
| Phase 8 / G6 | Module/file plan, environment strategy, development plan, TD-001/HG-001, and blueprint artifacts link design to executable work. |
| Phase 9 | Implementation records, code reviews, commits/PRs, and known defects link tasks/design to code and initial tests. |
| Phase 10 / G7 | Test strategy, acceptance scenarios, QA results, bug reports, and validation sign-off link requirements/features/code to verification evidence. |
| Phase 11 / G8 | Release readiness, known issues, release notes, rollback plan, security review, and final approval link validation evidence to release approval. |
| Phase 12 / G9 | Deployment checklist, deployment record, migration record, environment configuration, and rollback confirmation link release approval to deployed state. |
| Phase 13 | Maintenance plan, incidents, maintenance changes, metrics, backlog, and patch records link operational reality to support and improvement decisions. |
| Phase 14 / G10 | Post-release review, lessons learned, recommendations, maintenance backlog, and process notes link outcomes to the next planning cycle. |

## Artifact Link Expectations

| Artifact type | Should link to |
| --- | --- |
| Gate decision | Required evidence, approver, decision outcome, follow-up action. |
| Requirement | Source need/problem, acceptance criteria, design, tests, implementation, defects. |
| Feature | Requirements, scope boundary, design, tasks, tests, release notes. |
| ADR / design artifact | Requirements/features, constraints, alternatives, affected modules/APIs/data/UI. |
| Task / implementation record | Requirement/feature, design artifact, source file(s), test(s), PR/review, defects. |
| Test / QA result | Requirement/feature, test case/scenario, build, defect, validation sign-off. |
| Defect / incident | Requirement/feature if applicable, release/deployment, affected component, fix, verification. |
| Release / deployment record | G8 approval, release notes, rollback plan, artifact/build, environment, validation, incident/rollback status. |
| Maintenance backlog item | Incident/metric/customer feedback/change request, owner, priority, follow-up release or recommendation. |
| Improvement recommendation | Lesson/metric/incident/backlog item, owner, next-cycle routing, process/change-control link. |

## Machine-Readable Evidence and Requirements ↔ Tasks

**Classification:** Keep — machine-readable traceability guidance for requirements, tasks, standards evidence, gates, and verification artifacts.

When programs use **TD-001** (`Agnostic Execution Decomposition — Create Tasks List Procedure.md`) with **`tasks_board.json`**, you may add **direct requirement links** on each task so automation can validate coverage without relying only on spreadsheets.

### Goals

1. **Traceability:** From any requirement ID, list linked tasks; from any task ID, list covered requirements.
2. **Coverage:** Flag requirements with **no** linked tasks (gaps) and tasks with **no** requirement (**orphans**), unless explicitly tagged not applicable with a recorded reason.
3. **Stability:** Immutable string IDs for requirements and tasks; document any rename or migration if IDs change.
4. **Automation:** A **report** command (script or CI step) that validates consistency and prints a summary—no silent drift.

### Deliverables (adapt paths to the repo)

| Artifact | Role |
| --- | --- |
| **Requirements catalog** | Single source of truth: at minimum `id`, `title`, `status` (e.g. draft/approved/deprecated), optional `priority`, `source`, `notes`. Prefer one structured file (JSON, YAML, or CSV) under version control. |
| **Tasks catalog** | Tasks with `id`, `title`, `status`, optional `iteration` / `phase`, and **`linked_requirement_ids`** (array of requirement IDs). |
| **Linking rules** | Documented cardinality (e.g. each task ≥1 requirement unless `requirements_not_applicable: true` with reason); how epics/stories vs technical chores are handled. |
| **Traceability report** | Read-only checker: load requirements + tasks; validate ID formats and references (no dangling IDs); list orphans and uncovered requirements; exit non-zero on violation **if** CI gating is chosen—otherwise exit 0 with warnings (pick one policy and document it). |
| **Standards applicability record** | Appendix D JSON/YAML copy where automation is used; links applicable CYBERCUBE standards to gates, owners, and evidence artifacts. |
| **Identifier conformance notes** | Evidence that lifecycle-local IDs, Namespace G IDs, CC-PIDs, and Namespace M names are not mixed and that public entity IDs do not expose raw database keys. |
| **Contributor docs** | Short README or CONTRIBUTING section: how to add a requirement, add a task, link them, run the report locally and in CI. |

### Constraints

- Do **not** invent requirements from tasks without explicit approval text in-repo or from stakeholders.
- Prefer **plain** formats and scripts runnable without mandatory cloud services.
- Keep **version one minimal**; optional fields and diagrams can follow.
- **Match** repository conventions (package manager, `scripts/`, naming).

### Acceptance criteria (procedure-level)

- Requirements and tasks live in **version-controlled** artifacts with **stable IDs**.
- Every **new task template** shows where **`linked_requirement_ids`** belong.
- Running the traceability **report** completes locally with clear output.
- Docs explain linking rules and how to resolve orphans and gaps.

### Phase mapping

- **Phase 6 — Planning and Scope Control:** baseline requirements catalog and task linking policy alongside `tasks_list.md` / `tasks_board.json`.
- **Phase 10 — Testing and Validation:** tie verification evidence (test results, reports) to the same requirement IDs where tasks already link—consistent with USSM §7.4 and Annex A matrices.
- **Phase 11 / G8 and Phase 14 / G10:** standards applicability evidence can be checked against Appendix D's machine-readable record where automation is used.
