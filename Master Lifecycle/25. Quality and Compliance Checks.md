# Quality and Compliance Checks

## 1. Purpose

This document maps quality and compliance checks across the Master Lifecycle. It does not replace phase documents, decision gates, required artifacts, or detailed standards; it identifies what must be checked before a project advances, releases, deploys, or closes the improvement loop.

Use this file with:

- **`21. Decision Gates.md`** — gate evidence and decisions.
- **`22. Required Documents.md`** — lifecycle artifact register.
- **`23. Project Complexity Levels.md`** — scaling depth and criticality overrides.
- **`24. Traceability Rules.md`** — end-to-end traceability expectations.
- **`28. Appendix A — Template Library.md`** — evidence templates.

---

## 2. Canonical Checklists

Formal review checklists for requirements, design, and testing are specified in **USSM Section 10.5 — Annex E — Compliance Checklists** inside `USSM — Unified Software Standards Manual v1.0.md`:

- **§10.5.1** — SRS (IEEE 29148 alignment)
- **§10.5.2** — Design / DDS (IEEE 1016 alignment)
- **§10.5.3** — Testing (IEEE 29119 alignment)

Use these as **default review gates** unless a project’s regulatory context mandates additional checks.

For **visual, interaction, and experiential** acceptance criteria beyond Annex E’s document-centric checks, use **`29. Appendix B — Checklists.md`** during Phase 7 design review and Phase 10 validation. Map findings to defects or requirement updates under normal change control.

## 4. Lifecycle Quality Map

| Phase / Gate | Minimum quality and compliance checks | Evidence source |
| --- | --- | --- |
| Phase 1 / G1 | Idea is clear enough for analysis; sponsor/context recorded; complexity estimate present; obvious security/privacy/compliance unknowns captured. | Idea Capture Form (Template A-0). |
| Phase 2 / G2 | Problem is validated by independent evidence; impacted users and consequences are clear; problem/solution confusion is corrected. | Problem Definition Document (Template A-0.1). |
| Phases 3–4 / G3 | Selection rationale, feasibility, business case, assumptions, risks, and funding decision are documented; high-risk unknowns have owners. | Templates A-3.1, A-3.2, A-3.3, A-4, A-5 where used. |
| Phase 5 / G4 | Requirements are testable, traceable, scoped, and reviewed; NFRs, acceptance criteria, and stakeholder/user needs are covered. | Requirements package (A-8), CRS (A-1), SRS (A-2), NFR (A-10 / NFR-001), USSM Annex E §10.5.1. |
| Phase 6 | Scope is controlled; features map to requirements; delivery plan has owners, milestones, dependencies, and traceability hooks. | Scope Document (A-6), Feature Inventory (A-9), TD-001/HG-001 where used. |
| Phase 7 / G5 | Architecture satisfies requirements and NFRs; ADRs capture material decisions; data/API/UI/module artifacts exist where applicable; accessibility and UX checks are planned. | ARD-001, ADRs, A-11, A-12, UXD-001, MDM-001, USSM Annex E §10.5.2, Appendix B. |
| Phase 8 / G6 | Environment, branching, module/file plan, development plan, setup guidance, coding standards, and initial test setup are ready. | Templates A-13, A-14, A-15, Phase 8 DGP-001 guidance, TD-001/HG-001 where used. |
| Phase 9 | Code follows approved design, secure coding, traceability, review, and Definition of Done; defects/debt are visible. | Implementation records, code review evidence, tests, traceability updates, known defects/debt. |
| Phase 10 / G7 | Test strategy executed; acceptance scenarios pass or exceptions are recorded; defects have severity/disposition; validation sign-off is recorded. | Templates A-16 through A-20, USSM Annex E §10.5.3, Appendix B where UI/UX applies. |
| Phase 11 / G8 | Release readiness, known issues, release notes, rollback plan, security review, and final approval are complete. | Templates A-21 through A-26 and A-25.1 where security review applies. |
| Phase 12 / G9 | Deployment is executed from approved release evidence; migration/configuration/rollback status and PDV are recorded. | Templates A-27 through A-31, USSM §8. |
| Phase 13 | Maintenance ownership, incidents, changes, backlog, metrics/actuals, and patch records are maintained and reviewed. | Templates A-32 through A-37, REF-001 where refactoring applies. |
| Phase 14 / G10 | Post-release review closes the loop; lessons, recommendations, backlog, and process improvements are routed. | Templates A-35 and A-38 through A-41, `27. Process Improvement.md`. |

## 5. CYBERCUBE Standards Applicability Matrix

Every project must determine which CYBERCUBE standards apply based on product type, data handled, integrations, security surface, criticality, deployment model, vendors, and AI usage. Applicability must be reviewed early, converted into requirements and design controls where applicable, and confirmed before release approval.

Use **`31. Appendix D — CYBERCUBE Standards Applicability Matrix.md`** as the reusable project-level evidence form and optional machine-readable record.

If a standard is marked **Conditional** or **No**, the project must record a brief non-applicability rationale. If applicability changes later, update the matrix and route any scope, requirement, design, testing, or release impact through normal change control.

| --- | --- | --- | --- | --- | --- | --- |
| PRCS Product Classification | Candidate production product, existing product extension, architectural facet change, or pre-product/cross-cutting work needing Work Type Tags | Yes | 1, 3–4, 11 | Candidate PRCS metadata at G3; approved `PRD-XXX` + `PCL-L.D.E.C` before production release at G8; Work Type Tag rationale where no PRD applies | G3 / G8 | Product Owner |
| Naming & Identifier Standard (`STD-ENG-001`) | Any project producing artifacts, public IDs, governance IDs, modules, components, files, APIs, logs, or release evidence | Yes | 5–11 | Stable artifact IDs; Namespace G governance IDs where applicable; CC-PID use for public entity IDs; no raw database IDs exposed; Namespace M module/component/file naming evidence | G6 / G8 / G10 | Engineering Lead |

### Applicability Reviewed Checkpoints

| Lifecycle point | Required check |
| --- | --- |
| Phase 4 | Identify high-risk applicable standards during feasibility and business case review. |
| Phase 5 | Convert applicable standards into requirements, NFRs, constraints, and acceptance criteria. |
| Phase 7 | Reflect applicable standards in architecture, security, data, API, integration, and UI design where relevant. |
| Phase 8 | Prepare tooling, environment, coding, testing, CI, and review controls needed to satisfy applicable standards. |
| Phase 11 | Confirm required CYBERCUBE standards evidence is complete before release approval. |
| Phase 14 | Review applicability gaps, evidence gaps, exceptions, and improvement actions for the next cycle. |

The project-level matrix is complete when Appendix D completion criteria are satisfied and the record is reviewed before G8 and again during G10.

## 6. Cross-Cutting Checks

Apply these checks at every phase where relevant:

| Check | Expectation |
| --- | --- |
| Traceability | Evidence links upstream and downstream per `24. Traceability Rules.md`; gate decisions cite required artifacts. |
| Security / privacy | Sensitive data, auth/authz, secrets, tenancy, security review, privacy, and compliance impacts are assessed at the appropriate depth. |
| Accessibility / UX | User-facing work applies Appendix B and UXD-001 where UI/UX is in scope. |
| Risk / waiver handling | Material risks, exceptions, deviations, and waivers are explicit, approved, time-bound where applicable, and routed through the correct governance path. |
| Change control | Scope-affecting or release-impacting changes after baseline flow through `26. Change Control.md`. |
| Evidence completeness | Required artifacts have owner, date/version, decision, approval, and follow-up actions where applicable. |
| Complexity scaling | Review depth follows `23. Project Complexity Levels.md`; criticality can increase rigor even when complexity is small. |
| CYBERCUBE standards applicability | Applicable standards are identified, evidenced, or explicitly marked not applicable with rationale per §5. |

## 7. Relationship to Phase Gates

Checklist completion supports, but does not replace, gate decisions. Each gate must still answer:

1. Can the project move forward?
2. What evidence proves it?
3. Who approves it?
4. What happens if it fails?

Use **`21. Decision Gates.md`** as the authority for gate outcomes and required evidence.
