# Process Improvement

## 1. Purpose

Process Improvement defines how lessons, findings, recommendations, failed-change reviews, incidents, audit gaps, and post-release observations are routed into durable improvements to the lifecycle, standards, templates, tooling, training, and delivery practices.

This procedure closes the loop after **G10 — Maintenance Review Completed** and also supports improvement actions discovered during any earlier phase.

## 2. When to Use This Procedure

Use Process Improvement when a finding affects how CYBERCUBE plans, builds, tests, releases, deploys, maintains, or governs software.

Typical triggers:

- G10 follow-up actions from **`20. Phase 14 — Post-Release Review.md`**.
- Lessons Learned Record (Template A-39).
- Improvement Recommendations (Template A-40).
- Process Improvement Notes (Template A-41).
- Failed or emergency changes from **`26. Change Control.md`**.
- Incident / Production Issue Records (Template A-33).
- Operational Metrics / Actuals Review (Template A-36).
- Audit findings, compliance gaps, repeated defects, recurring support issues, or repeated gate failures.

## 3. Improvement Types

| Type | Examples | Typical owner |
| --- | --- | --- |
| **Lifecycle process** | Phase sequencing, gate criteria, handoff rules, review cadence. | Process owner / governance reviewer |
| **Template / documentation** | Appendix A templates, required-document entries, phase procedures. | Documentation owner / phase owner |
| **Engineering practice** | Coding standards, testing approach, review rules, CI/CD gates. | Engineering lead |
| **Operations / SRE** | Monitoring, alerting, rollback, deployment, incident handling. | Operations / SRE lead |
| **Security / privacy / compliance** | Security review gaps, data handling, audit evidence, waiver process. | Security / compliance owner |
| **Training / enablement** | Onboarding, reviewer guidance, team checklists, examples. | Team lead / enablement owner |

## 4. Improvement Intake

Each improvement candidate should record:

```text
Improvement ID:
Source:
Source Artifact Link:
Description:
Problem / Opportunity:
Affected Phase(s):
Affected Document(s) / Template(s):
Owner:
Priority:
Target Date:
```

Recommended ID format:

```text
PI-YYYY-NNN
```

Use Template A-41 when the improvement affects lifecycle process, standards, tooling, handoffs, or governance. Use Template A-40 when the improvement is primarily a product, operational, support, or backlog recommendation.

## 5. Triage and Prioritization

Prioritize improvements by:

- risk reduction,
- recurrence prevention,
- customer or user impact,
- compliance/audit impact,
- engineering efficiency,
- operational reliability,
- template or process reuse,
- effort and adoption cost.

| Priority | Treatment |
| --- | --- |
| **P0** | Immediate action; may block release, deployment, or next-cycle planning. |
| **P1** | Schedule in the next active planning or governance cycle. |
| **P2** | Track in the improvement backlog with an owner and target review date. |
| **P3** | Keep as observation or optional enhancement. |

## 6. Routing Rules

| Improvement affects | Route to |
| --- | --- |
| Product scope, requirements, feature priority | Product backlog / planning and `26. Change Control.md` when baselined scope changes. |
| Lifecycle phase, gate, template, or required document | Process owner and update to relevant Master Lifecycle file(s). |
| Architecture or technology decision | Architecture governance / ADR update. |
| Security, privacy, compliance, or risk posture | Security/compliance owner and risk register where required. |
| Release, deployment, rollback, or operational readiness | Release/operations owner and change control where release posture changes. |
| Test strategy, CI/CD, quality gate, or coverage rule | QA/engineering owner and quality/compliance review. |
| Training or team behavior | Team lead / enablement owner. |

## 7. Approval Rules

- Minor wording or clarification improvements may be approved by the relevant document owner.
- Changes to gate evidence, mandatory artifacts, lifecycle sequencing, security/privacy controls, or release/deployment requirements require the relevant owner and governance reviewer.
- Improvements that change approved scope, release posture, operational commitments, or production behavior must flow through **`26. Change Control.md`**.
- Permanent exceptions are not process improvements; they require an approved waiver or standards change.

## 8. Implementation and Verification

For each approved process improvement:

1. Update the affected document, template, checklist, script, or workflow.
2. Record before/after summary and rationale.
3. Update related references, especially `21. Decision Gates.md`, `22. Required Documents.md`, and `28. Appendix A — Template Library.md` where applicable.
4. Communicate the change to affected owners or teams.
5. Verify the change resolved the original finding.
6. Record any follow-up action or adoption review date.

## 9. Improvement Backlog

Maintain improvement candidates in an improvement backlog or equivalent tracker. Each item should include:

- stable ID,
- source artifact,
- owner,
- priority,
- status,
- due date or review date,
- affected lifecycle area,
- decision or approval,
- closure evidence.

Suggested statuses:

```text
New -> Triaged -> Approved -> In Progress -> Implemented -> Verified -> Closed
```

Rejected or deferred items must keep rationale and re-review conditions.

## 10. Quality Checks

- Improvement has a stable ID and source evidence.
- Affected phase, gate, document, template, or owner is identified.
- Priority and routing are clear.
- Required approvals are recorded.
- Related documents are updated consistently.
- Change control is used when the improvement changes approved scope, release posture, operational commitments, or production behavior.
- Closure includes verification that the original issue or opportunity was addressed.
