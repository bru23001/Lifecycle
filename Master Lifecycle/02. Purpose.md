# Purpose

This section defines why the Master Lifecycle procedure exists, who shall use it, what decisions it controls, and what evidence it expects. It complements **`01. Introduction.md`**, which defines the procedure family and identifies **USSM v1.0** as the SDLC documentation spine.

Use **`03. Scope.md`** for applicability and lifecycle boundaries. Use **`04. Definitions.md`** for controlled vocabulary. Use **`05. Roles and Responsibilities.md`** for ownership and handoffs. Use **`06. Lifecycle Overview.md`** for the fourteen-phase structure, approval gates, USSM mapping, and documentation tiers. Use **`21. Decision Gates.md`** for gate criteria and continuation rules. Use **`22. Required Documents.md`** as the maintained artifact register.

---

## 1.1 Purpose Statement

This procedure defines the complete lifecycle for selecting, planning, designing, developing, testing, releasing, operating, and maintaining software projects.

It provides a repeatable and governed process to ensure that every project is valuable, feasible, well-documented, secure, testable, maintainable, and aligned with business goals.

The procedure prevents software projects from being started casually, planned incompletely, developed inconsistently, or released without sufficient evidence of readiness. It establishes a structured path from the first project idea through long-term maintenance and improvement.

This procedure also ensures that each project produces clear decisions, required documentation, traceable requirements, quality evidence, and approval gates before moving forward.

---

## 1.2 Why This Process Exists

This process exists to create a consistent method for managing software projects from beginning to end.

It is intended to solve the following problems:

* unclear project goals
* weak problem definition
* poor project selection
* incomplete requirements
* missing architecture decisions
* undocumented design choices
* uncontrolled scope changes
* weak testing coverage
* rushed releases
* poor maintenance planning
* lack of traceability between requirements, code, tests, and release decisions

The process ensures that software projects are not only built, but built correctly, reviewed properly, released safely, and improved over time.

This aligns with the CYBERCUBE principle that standards and procedures shall be clear, practical, current, enforceable, and traceable.

---

## 1.3 Who Will Use This Procedure

This procedure is intended for anyone involved in selecting, planning, building, reviewing, releasing, operating, or maintaining software projects.

Primary users include:

* project owners
* software developers
* architects
* UI/UX designers
* testers and QA reviewers
* product owners
* technical leads
* security reviewers
* operations or DevOps personnel
* maintainers
* business stakeholders
* future team members who need to understand how a project was created

For small solo projects, one person may perform several of these roles. For larger projects, responsibilities may be divided across a team.

Accountability by function and typical lifecycle handoffs are defined in **`05. Roles and Responsibilities.md`**.

---

## 1.4 What Kinds of Projects This Procedure Applies To

This procedure applies to software projects of different sizes and complexity levels, including:

* websites
* web applications
* mobile applications
* desktop applications
* SaaS platforms
* internal business tools
* automation systems
* APIs and backend services
* dashboards and data systems
* AI-assisted software systems
* reusable modules
* customer portals
* MVPs and prototypes intended to become real products

The procedure shall be scaled according to project complexity. A small website does not require the same documentation burden as a regulated SaaS platform, but both shall follow the same basic lifecycle.

## 1.5 What Decisions This Procedure Controls

This procedure controls the major decisions that determine whether a project should continue, change direction, pause, or stop.

| Decision Area | Controlled Question |
|---|---|
| Project selection | Should this project be pursued? |
| Problem validation | Is the problem real and worth solving? |
| Business value | Does the project deliver measurable benefit? |
| Feasibility | Can the project be built with available time, skill, budget, and technology? |
| Scope | What is included and excluded? |
| Requirements | What must the system do? |
| Architecture | How should the system be structured? |
| Technology stack | What tools, languages, frameworks, and platforms should be used? |
| Design readiness | Is the project ready for development? |
| Test readiness | Is the project testable and verifiable? |
| Release readiness | Is the project safe and complete enough to release? |
| Maintenance readiness | Can the project be supported after release? |
| Change control | Should a requested change be accepted, deferred, or rejected? |

Technology decisions shall follow authoritative governance instead of being chosen ad hoc. Selection shall align with the **Technology Radar** (Adopt / Trial / Assess / Hold), controlled deviations, and ADR discipline described in **`../CYBERCUBE standards/[3]-POL-GOV-002 CYBERCUBE-Architecture-Governance-Policy-v1.md`** (Architecture Governance Policy). Material choices are recorded per that policy and **`13. Phase 7 — Architecture and Design.md`**.

## 1.6 What Documents Must Be Produced

This procedure requires project documentation that proves the project was selected, planned, designed, developed, tested, released, and maintained in a controlled way.

The living register of representative artifacts, templates, and traceability expectations is **`22. Required Documents.md`**, aligned to USSM tiers. At minimum, the process should produce the following document categories:

| Category | Example Documents |
|---|---|
| Project definition | Idea Capture Form, Problem Definition Document |
| Selection and approval | Project Selection Scorecard, Feasibility Assessment, Business Case |
| Requirements | Customer Requirements Specification, Software Requirements Specification, Non-Functional Requirements |
| Planning | Scope Document, Feature Inventory, Roadmap, Iteration Plan |
| Architecture and design | Architecture Document, Module Design, Component Design, Data Model, API Contract |
| UI/UX | User Personas, User Journeys, Wireframes, Screen Inventory |
| Development preparation | Directory Structure, File Specification Catalog, Environment Strategy |
| Testing | Test Strategy, Acceptance Test Scenarios, QA Checklist |
| Release | Release Readiness Checklist, Deployment Checklist, Rollback Plan |
| Maintenance | Maintenance Plan, Incident Response Notes, Post-Release Review |

CYBERCUBE documentation guidance separates document types such as standards, guides, RFCs, ADRs, runbooks, references, and tutorials so that each document has a clear purpose and audience.

## 1.7 Expected Outcome

By following this procedure, every software project should have:

* a clearly defined problem
* a justified reason for being selected
* documented requirements
* controlled scope
* approved architecture
* traceable design decisions
* testable features
* release evidence
* deployment preparation
* maintenance planning
* post-release improvement feedback

The final outcome is a repeatable lifecycle that helps CYBERCUBE build software projects with consistency, accountability, and long-term maintainability.
