# Appendix C — Scoring Models

## 1. Purpose

This appendix collects scoring models used by the Master Lifecycle. Scores support decisions; they do not replace gate approvals, required evidence, named accountability, or professional judgment.

Use scoring to:

- compare candidate initiatives,
- classify complexity and review depth,
- prioritize scope, features, risks, defects, changes, and improvements,
- support release readiness and compliance decisions,
- make tradeoffs explicit and auditable.

## 2. Scoring Principles

- Define criteria before scoring.
- Record evidence links for each material score.
- Use the same scale consistently within one decision.
- Document weights when criteria are not equal.
- Treat unknowns as risk, not as zero.
- Preserve reviewer, date, rationale, and decision outcome.
- Use higher rigor when complexity, criticality, data sensitivity, security, privacy, compliance, or operational exposure requires it.

## 3. Common Scale

Use a 1–5 scale unless a template or program defines another scale.

| Score | Meaning |
| --- | --- |
| 1 | Weak / low fit / high concern |
| 2 | Limited / below expectations |
| 3 | Acceptable / moderate / baseline |
| 4 | Strong / low concern |
| 5 | Excellent / high confidence |

For risk, use likelihood × impact where appropriate:

| Risk score | Severity |
| --- | --- |
| 1–4 | Low |
| 5–9 | Medium |
| 10–16 | High |
| 17–25 | Critical |

## 4. Project Selection Scoring

Use **Template A-3.1 — Project Selection Scorecard** for Phase 3 selection. Criteria may include:

| Criterion | Typical weight | Notes |
| --- | --- | --- |
| Strategic alignment | 10–20% | Fit with product, customer, operational, or compliance goals. |
| Customer / user value | 10–20% | Pain severity, reach, usability or support value. |
| Business value | 10–20% | Revenue, savings, retention, risk reduction, efficiency. |
| Feasibility | 10–20% | Technical, staffing, schedule, dependency confidence. |
| Risk / compliance exposure | 10–20% | Lower risk scores higher unless risk reduction is the goal. |
| Urgency / timing | 5–15% | Deadline, market window, regulatory or operational pressure. |
| Maintainability / operational fit | 5–15% | Support cost, observability, ownership, lifecycle fit. |

Recommended formula:

```text
Weighted Score = Sum(score x weight)
```

The scorecard is a selection signal only. **G3 — Project Selected** is not complete until Phase 4 adds feasibility and business-case evidence.

## 5. Complexity and Criticality Scoring

Use **`23. Project Complexity Levels.md`** for the canonical Small / Medium / Large / Unknown rubric.

Scoring can help assign the level:

| Driver | Low / Small | Medium | Large / High |
| --- | --- | --- | --- |
| Users / reach | Few internal users | Multiple groups | Broad customer or tenant reach |
| Data sensitivity | Public/internal only | Some confidential data | PII, regulated, restricted, tenant-sensitive |
| Integrations | None or simple | Several known integrations | Complex, external, business-critical integrations |
| Availability | Low impact downtime | Moderate operational impact | High availability or SLA/SLO commitments |
| Security / compliance | Basic controls | Security/privacy review needed | Regulated, authz, tenant isolation, audit-heavy |
| Delivery uncertainty | Known path | Some unknowns | Major unknowns or high dependency risk |

Criticality overrides complexity. A small change can require high rigor if it affects regulated data, security controls, auth/authz, production infrastructure, customer commitments, or release posture.

## 6. MoSCoW / Priority Scoring

Use for scope, feature inventory, maintenance backlog, improvement recommendations, and release decisions.

| Priority | Meaning |
| --- | --- |
| Must | Required for gate, release, compliance, safety, or committed scope. |
| Should | Important but can be deferred with accepted impact. |
| Could | Valuable if capacity allows. |
| Won't / Not now | Explicitly out of current scope or deferred. |

Alternative priority codes:

| Code | Meaning |
| --- | --- |
| P0 | Blocks release, deployment, support, or critical operation. |
| P1 | Needed in current cycle. |
| P2 | Planned follow-up. |
| P3 | Optional / observation. |

## 7. Change and Defect Risk Scoring

Use likelihood × impact for change requests, defects, incidents, and release blockers.

| Impact factor | Considerations |
| --- | --- |
| Customer impact | Number of users, severity of workflow disruption, support burden. |
| Data impact | Loss, corruption, privacy, retention, tenant isolation, migration risk. |
| Security impact | Auth/authz, secrets, vulnerabilities, exposure, abuse potential. |
| Operational impact | Availability, rollback difficulty, observability, incident likelihood. |
| Compliance impact | Contractual, regulatory, audit, or policy obligations. |

Risk treatment should align with **`26. Change Control.md`** and relevant gate evidence.

## 8. Testing and Release Readiness Scoring

For **G7** and **G8**, scores should support pass/fail/conditional decisions.

| Area | Example signal |
| --- | --- |
| Test execution | Planned vs run vs passed, blocked tests, waived tests. |
| Defects | Open critical/high count, deferred defects, verification status. |
| Coverage | Requirement coverage, unit/integration/system/acceptance coverage. |
| CI/CD health | Build stability, static analysis, security/accessibility checks. |
| Release readiness | Known issues, rollback confidence, support readiness, security review. |

Release should not be approved on aggregate score alone when critical blockers exist.

## 9. UI/UX Scoring

Use **`29. Appendix B — Checklists.md`** for UI/UX scoring. Common dimensions include:

- visual hierarchy,
- consistency,
- feedback and states,
- typography/readability,
- accessibility,
- information architecture,
- usability heuristics,
- task friction.

UI/UX findings should map to requirements, defects, or improvement recommendations.

## 10. CYBERCUBE Release / Compliance Scoring

When product classification, criticality, or release policy requires CYBERCUBE release scoring, use the CYBERCUBE compliance matrix:

- 9 dimensions, 0–5 each, maximum 45.
- **>= 40**: Release-grade.
- **32–39**: Conditional; remediation required.
- **< 32**: Blocked.

Dimensions:

| Dimension | Focus |
| --- | --- |
| Content | Clarity, terminology, versioned copy. |
| Yield | KPIs, measurable value, post-launch validation. |
| Brand | Voice, tone, naming, visual rules. |
| Engagement | CTAs, progressive disclosure, interaction signals. |
| Responsiveness | Multi-device, accessibility, performance. |
| Conversion | Funnels, forms, drop-off analysis. |
| UX | IA, recovery paths, research-based usability. |
| Backend | Security, scalability, observability. |
| Experience | Trust, onboarding/offboarding, support transparency. |

This score can support **G8 — Release Approved** but does not replace required release evidence.
