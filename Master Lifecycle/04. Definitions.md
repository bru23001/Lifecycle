# Definitions

Controlled terms used in **`01. Introduction.md`**, **`02. Purpose.md`**, **`03. Scope.md`**, and the numbered phase documents **`07.`–`20.`** Unless a phase file defines a term differently for that phase, these meanings apply.

| Term | Definition |
| --- | --- |
| **Approval / decision gate** | A defined point where the project shall not proceed to the next lifecycle phase (or major milestone) until required evidence exists and the applicable approval or decision has been recorded. Criteria and artifacts are detailed in **`21. Decision Gates.md`** and the relevant phase document. |
| **Artifact ID** | A stable identifier assigned to a lifecycle artifact, record, or evidence item so it can be cited in gates, traceability matrices, tasks, tests, releases, and audits. Artifact IDs are lifecycle-local unless registered as CYBERCUBE governance IDs. |
| **CC-PID** | CYBERCUBE public entity identifier defined by `STD-ENG-001A`; used for public/external references to entities in APIs, URLs, logs, support workflows, or UI where an external-safe identifier is needed. CC-PIDs must not be used as authentication or authorization proof. |
| **CRS** | **Customer Requirements Specification** — stakeholder needs, constraints, and success criteria (USSM documentation tier 1). |
| **CYBERCUBE software work** | Software delivery or evolution performed under CYBERCUBE governance: new development, changes to existing products, or project-style initiatives that produce or modify software. |
| **DDS / SDD** | **Design Documentation** / **Software Design Description** — architecture, interfaces, data, APIs, and design baselines (USSM documentation tier 3). |
| **Master Lifecycle** | This procedure family: modular documents that together define CYBERCUBE’s full software lifecycle from idea capture through post-release review. |
| **Namespace A** | CYBERCUBE artifact-naming namespace for mutable artifacts such as files, packages, branches, and deliverables. |
| **Namespace B** | CYBERCUBE public entity ID namespace for immutable CC-PID identifiers. |
| **Namespace G** | CYBERCUBE governance registry namespace for immutable sequential governance IDs such as `PRD-XXX`, `MOD-XXX`, and future registered governance identifiers. |
| **Namespace M** | CYBERCUBE module, component, and file naming namespace for source-code modules, components, directories, exports, and implementation files. |
| **Normative** | Binding for compliance; “shall” / “must” expectations that teams are expected to meet or explicitly tailor per governance. |
| **Overlay manual / overlay** | A domain-specific procedure, blueprint, or standard that **adds** rules or patterns on top of the Master Lifecycle. Overlays **do not replace** lifecycle phases or gates. |
| **Procedure family** | The set of related lifecycle documents (introduction, purpose, scope, overview, phases **`07.`–`20.`**, appendices, and registers) that share the same USSM-aligned backbone. |
| **Product evolution** | Changes to an existing product line (features, fixes, refactors) managed with the same lifecycle principles as a greenfield **project**, with scaling applied to documentation depth. |
| **Project** | A bounded initiative that delivers or materially changes software, with a defined purpose, scope, and lifecycle progression through the fourteen phases. |
| **Required outputs** | Artifacts, records, or decisions that a phase document and **`22. Required Documents.md`** require before the phase is complete or before passing the next gate. |
| **Scaling** | Applying the **same** lifecycle phases while adjusting **depth** of documentation, rigor of reviews, and extent of evidence based on risk, criticality, complexity, and regulatory need (see **`02. Purpose.md`** §**1.4** and **`23. Project Complexity Levels.md`** when used). |
| **SRS** | **Software Requirements Specification** — testable functional and non-functional requirements (USSM documentation tier 2). |
| **Sufficient evidence** | Enough validated information, completed outputs, and recorded decisions to justify continuing, pausing, redirecting, or stopping the project at a gate (see **`03. Scope.md`** §**3.5**). |
| **Tailoring** | A **documented** adjustment of how the lifecycle is applied to a specific initiative (depth, templates, or timing of artifacts) while remaining compliant with mandatory governance; must be traceable to an approved decision where required. |
| **Traceability** | Demonstrable links between needs, requirements, design, implementation, tests, releases, and maintenance changes so that impact and compliance can be verified (see **`24. Traceability Rules.md`**). |
| **Stable ID** | An identifier that remains consistent after approval or baseline so evidence links do not drift. If a stable ID must change, the migration from old ID to new ID must be recorded. |
| **USSM** | **Unified Software Standards Manual** (`USSM — Unified Software Standards Manual v1.0.md`) — normative documentation spine aligned with **ISO/IEC 12207** and **IEEE 29148**. |
| **Waiver / exception** | A **time-bound**, governance-approved departure from a normally applicable requirement, recorded per CYBERCUBE policy with risk acceptance where required (see **`../CYBERCUBE standards/[6]-STD-GOV-003 CYBERCUBE-Policy-Exception-Waiver-Standard-v1.md`**). |
