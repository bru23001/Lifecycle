# Change Control

## 1. Purpose

Change Control governs changes to baselined lifecycle artifacts, approved scope, release posture, production environments, maintenance commitments, and process outputs. It ensures changes are assessed, approved, documented, reversible where feasible, and traceable to affected requirements, designs, tests, releases, deployments, maintenance records, and improvement actions.

This procedure applies across the Master Lifecycle and aligns with CYBERCUBE Change Management Policy **5.7**.

## 2. When Change Control Applies

Use change control when a proposed change affects:

- approved scope, requirements, acceptance criteria, or feature inventory,
- architecture, data model, API/integration contract, UI/UX design, or module boundaries,
- release approval evidence, deployment procedure, rollback path, or production configuration,
- security, privacy, compliance, tenant/data isolation, or operational commitments,
- maintenance backlog priority, incident remediation, patch/release content, or post-release improvement routing,
- any gate decision or required evidence in **`21. Decision Gates.md`**.

Development work before baseline approval may use normal backlog refinement unless the phase document or project governance requires a formal change record.

## 3. Change Types

| Type | Use | Approval path |
| --- | --- | --- |
| **Standard change** | Low-risk, repeatable, pre-approved change with a known procedure. | Pre-approved catalog or team lead; log after execution. |
| **Normal change** | Planned change with material scope, technical, release, operational, or compliance impact. | Change owner submits request; approval by Engineering Lead, Product Owner, CAB, or sponsor depending on risk. |
| **Emergency change** | Urgent change needed for incident response, security threat, imminent outage, compliance deadline, or production failure. | Emergency approval by incident/change authority; formal record completed within 24 hours. |

Emergency change control is not a shortcut for poor planning. Repeated emergency changes for the same cause require root cause analysis and process improvement.

## 4. Risk Classification

| Risk | Typical indicators | Minimum treatment |
| --- | --- | --- |
| **Low** | Small impact, easy rollback, no sensitive data, no customer-visible disruption. | Basic assessment, documented test/verification, owner approval. |
| **Medium** | Some user/customer impact, configuration or dependency changes, moderate rollback or support needs. | Impact/risk assessment, testing evidence, rollback plan, named approver. |
| **High** | Many users/customers, production release impact, data migration, security/privacy impact, difficult rollback. | Full assessment, staging/validation evidence, rollback tested where feasible, CAB or equivalent review. |
| **Critical** | Core system, data loss risk, regulated impact, security incident, broad outage, irreversible change. | Executive/CAB approval, full evidence, incident/rollback readiness, post-implementation review. |

Complexity does not reduce risk treatment. Use **`23. Project Complexity Levels.md`** for scaling depth, and apply the higher rigor when product criticality, data sensitivity, or operational exposure demands it.

## 5. Change Request Record

Use a stable ID for each formal change request:

```text
CR-YYYY-NNNNN
```

The change request should include:

| Section | Required content |
| --- | --- |
| Change information | ID, title, requester, owner, date, affected product/release/environment. |
| Classification | Standard / Normal / Emergency; Low / Medium / High / Critical risk. |
| Description and rationale | What changes, why it is needed, and expected outcome. |
| Scope and affected artifacts | Requirements, features, design, tasks, code, tests, releases, deployments, maintenance records, or processes affected. |
| Impact assessment | User/customer impact, data/security/privacy/compliance impact, operational impact, dependencies, and timing. |
| Implementation plan | Pre-change, during-change, and post-change steps. |
| Test / validation plan | Evidence needed before and after implementation. |
| Rollback / mitigation plan | Trigger conditions, steps, owner, expected duration, data recovery notes. |
| Communications | Internal, customer, support, or operational communications. |
| Approval record | Approver, role/authority, decision, date, conditions. |
| Post-change review | Outcome, incidents, follow-ups, lessons, and process improvements. |

## 6. Lifecycle Gate Triggers

| Gate / phase | Change-control trigger |
| --- | --- |
| **G3 — Project Selected** | Funding, feasibility, project selection, or business-case assumptions materially change. |
| **G4 — Requirements Approved** | Baselined requirements, NFRs, acceptance criteria, or scope change. |
| **G5 — Architecture Approved** | Architecture, ADRs, data model, API/integration, UI/UX, or module boundaries change materially. |
| **G6 — Development Ready** | Environment strategy, module/file plan, development plan, branching, or initial test setup changes materially. |
| **G7 — Testing Passed** | Defects, waivers, validation scope, or acceptance evidence changes the test decision. |
| **G8 — Release Approved** | Known issues, release notes, rollback plan, security review, or final approval conditions change. |
| **G9 — Deployment Completed** | Deployment, migration, configuration, rollback, or operational handoff requires remediation or re-approval. |
| **G10 — Maintenance Review Completed** | Maintenance backlog, improvement recommendations, or process changes affect the next planning cycle or governance process. |

G1/G2 discovery changes are usually handled by revising the idea/problem records unless they affect downstream baselines.

## 7. Approval Rules

- The requester should not be the sole approver for their own material change.
- Security, privacy, compliance, infrastructure, or customer-impacting changes require the relevant reviewer.
- High/Critical changes require named approval and documented risk acceptance where residual risk remains.
- Emergency approval may be verbal or lightweight during response, but must be documented within 24 hours.
- Rejected or deferred changes must record rationale and re-review conditions.

## 8. Traceability and Evidence

Every approved change should link to affected lifecycle evidence. Use **`24. Traceability Rules.md`** and include links to:

- source requirement, feature, problem, defect, incident, or improvement recommendation,
- affected templates or artifacts in **`22. Required Documents.md`**,
- affected gate decision in **`21. Decision Gates.md`**,
- implementation records, tests, releases, deployments, or maintenance records,
- risk, waiver, or exception records where applicable.

Traceability must be updated after the change is implemented.

## 9. Implementation, Verification, and Rollback

Before implementation:

- confirm approval and change window,
- confirm test/validation plan,
- confirm rollback/mitigation path,
- confirm communications and support impact,
- confirm access and environment readiness.

After implementation:

- verify the intended outcome,
- record test/validation results,
- update affected artifacts and traceability links,
- record rollback confirmation or non-invocation where deployment is involved,
- open incident response if the change causes customer-impacting failure.

## 10. Post-Implementation Review

Post-implementation review is required for:

- failed changes,
- emergency changes,
- High/Critical changes,
- changes that caused incidents, customer impact, security findings, rollback, or material delay,
- repeated changes for the same root cause.

Review outcomes should feed **`27. Process Improvement.md`**, the Maintenance Backlog (Template A-35), Lessons Learned Record (Template A-39), Improvement Recommendations (Template A-40), or Process Improvement Notes (Template A-41) as applicable.

## 11. Quality Checks

- Change request has a stable `CR-` ID.
- Affected lifecycle artifacts and gates are identified.
- Risk level and change type are recorded.
- Testing and rollback/mitigation are documented.
- Required reviewers approved the change.
- Traceability links are updated after implementation.
- Failed/emergency/material changes have post-implementation review.
